# AI 线索分级处理系统 — 产品需求文档（PRD）
> 版本 1.0 | 2026年4月
> 客户：Xpatweb | 行业：移民咨询（南非）

---

## 1. 产品愿景

一套 AI 驱动的线索分级中间件，介于 Xpatweb 的入站渠道（邮件、网页表单、WhatsApp）与顾问之间。系统自动完成询盘资质判断、个性化回复起草、路由分发，并将所有信息记录到 CRM——将首次回复时间从数小时压缩到数分钟，同时消除目前 35% 的无效线索对顾问时间的浪费。

---

## 2. 问题陈述

| 问题 | 数据依据 |
|------|---------|
| 35% 的线索低价值或无价值 | 历史数据：202条线索中71条评为 Bad |
| 顾问反复手写相同邮件 | 20+ 个签证专属回复模板存在，但全部人工操作 |
| 评分标准不一致，依赖个人判断 | 评分标准以 PDF 形式存在，从未纳入工作流程 |
| 紧急/投诉客户可能被漏掉 | 规程要求升级给 Jerry，但完全靠人工识别 |
| 所有邮件需要双人审核 | "四眼原则"存在于规程中，但没有任何系统支撑 |

## 4. 核心功能

### 4.1 数据接入层
- **多品牌收件箱：** 4个品牌（XP、RISA、VLS、SMV）各有独立收件箱——每条线索必须打上 `source_box` 标签
- **邮件接入：** 每个收件箱通过 OAuth 连接 Gmail/Outlook；轮询或使用 webhook/邮件转发规则
- **网页表单 webhook：** 接收联系表单的 POST 请求（自动填充 Lead ID、日期、来源、邮箱、电话）
- **WhatsApp 接入（Phase 2）：** 通过 Twilio/WhatsApp Business API 接入 ML393 / Wati 来源的线索
- **手动录入：** 提供表单用于录入电话询盘和非表单来源的邮件/转介线索
- **线索来源标记：** 自动识别 UTM / 活动代码；保留 `lead_source`（XP349、RSA024 等）

### 4.2 AI 解析与提取
解析每条入站消息，提取结构化字段：

```
sender_name, email_domain, lead_type (Individual/Corporate),
visa_category, job_title, nationality, net_worth_indicator,
urgency_flag, multi_visa_flag, email_coherence, raw_message
```

模型：**Gemini Flash，temperature=0**（2026-06-04 实施决策）。使用 structured output / JSON Schema 保证字段 schema 一致性。

### 4.3 线索评分引擎
在提取字段基础上应用确定性评分标准（来自 Lead Score Sheet）：

- **规则预过滤：** DNQ 硬规则（6条否决条件，见 [业务规格.md §4.1](业务规格.md)）优先用 Python 函数判定——零 token、结果确定、可逐条测试；Python 只执行已确认的结构化事实，不负责理解自然语言
- **软规则标记：** 高风险条件（PBS 薪资、Visitor 11(6) 关系，见 [业务规格.md §4.2](业务规格.md)）设 `risk_flags` 字段，**不自动拒绝**
- **评分矩阵：** 未命中 DNQ 的线索 → 走 LLM 评分 → GD / MF / MD / BD
- **置信度标记：** DNQ 所需字段为 `null/unknown`、互相矛盾或提取置信度不足时，转人工补充/复核，**不得自动判为 DNQ**
- **评分解释：** 每条线索必须有简洁的白话评分理由（如"Gold：Retired Person Visa + Director 职称 + 措辞专业 + 有紧迫性"）

> **准确性边界：** Python 硬规则不会像 LLM 一样临场猜测，因此在规则和输入事实正确时更稳定；真正的误判风险主要来自上游字段提取错误、业务规则过期或边界条件定义不完整。系统采用“LLM 理解语言 → 数据校验 → Python 执行明确硬规则 → 不确定项转人工”的混合方案，并通过独立 DNQ fixture 同时验证命中与误杀情况。

### 4.4 回复起草
- **GD 线索：** 起草咨询预约邮件 + WhatsApp 语音消息话术 + 电话话术
- **MF/MD 线索：** 起草含7项服务介绍 + 费用 + 材料清单的完整报价邮件
- **BD 线索（无 DNQ 原因）：** 起草简短报价或要求补充信息邮件
- **BD 线索（命中 DNQ）：** 选择对应拒绝模板 + 如有替代签证则一并建议
- 填充个性化字段：姓名、品牌、签证类型、费用、行政费、顾问姓名、材料清单
- 品牌署名根据 `source_box` 变化（Xpatweb / Retire In South Africa / Visa Litigation Services）
- 输出：邮件草稿 + 供内部群发布的 WhatsApp 帖子文案

模型：**Gemini**（2026-06-04 实施决策，与 [业务规格.md §10.5](业务规格.md) 同步）。费用金额仍必须来自模板/配置，禁止由模型自由生成。

### 4.5 路由与分配
- **GD 线索：** 分配高级顾问；立即向 Leads 团队触发60秒致电提醒；生成电话话术
- **MF/MD 线索：** 触发60秒致电提醒；生成补充信息确认话术
- **BD 线索：** 不致电——直接将报价/DNQ 草稿送入审核队列
- **升级标记：** 立即通知 Jerry——绕过标准队列
- **DNQ/Bad 线索：** 拒绝类草稿进入 Marisa / 指定审核人确认路径后再发送
- **签证核查：** 标记给 Willem Pretorius
- **不同时区的 GD 线索：** 标记先发 WhatsApp 语音消息，再在对方工作时间致电
- **所有外发邮件：** 发送前进入四眼审核队列

### 4.6 四眼审核队列
- 仪表盘展示待审核草稿
- 审核人可操作：批准 / 编辑后批准 / 拒绝（退回修改）
- 外发邮件须经第二人检查后才能发送；拒绝邮件进入 Marisa / 指定审核人确认路径
- 完整审计记录：谁在何时批准

### 4.7 客户背景调研与研究驱动草稿（GD + MF 线索）

> **来源标记：**
> - 🟡【PM 建议 — 已与对接人内部对齐】整体功能形态：自动化背景调研 + 结构化简报 + 研究驱动草稿，给 Leads 团队使用
> - ✅【客户已确认】Sales Protocol 中明确 Leads 团队**人工** LinkedIn 查询，用于辅助评分和打电话前了解背景
> - ⬜【待客户确认】调研来源的具体范围（除 LinkedIn 外的 Google News、公司官网、CIPC 是否纳入）；Research Brief 的字段是否符合 Leads 团队实际工作流
>
> 客户方原始 4 份材料**没有明确要求**自动化调研模块。本节是基于「人工 LinkedIn 查询」的产品化扩展。

针对 GD 和 MF 线索，评分完成后立即触发异步调研任务。调研不阻塞60秒致电提醒——在后台运行，在咨询前准备就绪。

**调研来源：** ⬜【待客户确认范围】
- LinkedIn：个人主页（职称、资历、教育、所在地）+ 公司主页（规模、行业、国际业务）— 唯一✅【客户已确认】的来源
- 公司官网：服务、团队、业务布局 — 🟡【PM 建议】
- Google News：近12个月关于此人或公司的相关新闻 — 🟡【PM 建议】
- CIPC（南非公司注册）：适用于企业线索 — 🟡【PM 建议】

**输出 1 — Research Brief（结构化调研简报，1页）：**
- 个人档案：职位、资历、国籍、学历、LinkedIn URL
- 雇主信息：规模、行业、南非业务、国际运营
- 移民背景分析：最佳签证路径评估、资格风险标记、竞争对手风险信号
- 近期新闻：2-3条要点，或 "Nothing notable found"
- 调研置信度：high / medium / low

**输出 2 — Consultant Briefing Note（顾问简报，最多3条）：**
- Leads 团队成员致电前须了解的关键信息
- 建议的开场白
- 需要特别关注的信号（升级信号、风险、紧迫性）

**输出 3 — 研究驱动的个性化邮件草稿：**
- 根据线索的职业背景个性化（自然引用职位/公司背景）
- 若调研发现更合适的签证则调整推荐
- 替代通用模板填充，提供有背景感知的草稿
- 字数上限350词；南非正式英语风格；不提及曾做过调研

### 4.8 CRM 记录
- 线索进入时自动创建记录，写入所有提取字段
- 在每个工作流步骤更新状态（Received → Scored → Drafted → In Review → Sent / DNQ）
- 对接现有 CRM（待定：HubSpot / Salesforce / Google Sheets 备选）
- 追踪：每条线索预估营收、签证类别分布、顾问工作量

## 5. 非功能性需求

> **重要说明：** 本节所有数值均为**衍生目标 / 行业基准 / 推荐值**，**没有任何一项**直接来自4份原始客户材料。客户原始材料只明确了一个时效约束：**4 小时回复 SLA + 60 秒致电规则**（见 [业务规格.md §6.5](业务规格.md)）。
>
> 下表所有目标在与客户对齐前不得对外承诺。

| 需求项 | 目标值 | 来源标记 |
|--------|--------|---------|
| 邮件到达至草稿就绪时间 | < 2 分钟 | ⬜【衍生目标】客户原始材料只要求 4 小时回复 SLA；2 分钟是为支撑 60 秒致电的衍生目标，待客户确认是否过严 |
| 评分与人工标准的一致率 | > 85% | ⬜【推荐基准】客户原始材料未提及此指标；85% 是行业经验值，需客户确认是否可接受及如何验收 |
| DNQ 规则命中准确率 | 95%+ | ⬜【推荐基准】在独立 DNQ fixture 上验证；系统只进入拒绝审核路径，不自动发送拒绝邮件 |
| 系统可用性 | 99.5% | ⬜【行业基准】原始材料未提及；待客户确认 |
| 数据存储位置 | 南非或欧盟（POPIA 合规） | ⬜【法务推断】南非业务存在 POPIA，但客户未明确合规要求；待客户法务确认 |
| 个人信息处理 | 邮件内容静态加密；不用于训练模型 | 🟡【PM 标准做法】Anthropic API 默认满足；建议在合同条款中明确 |

## 7. 交付范围（Phase 0 / Phase 1 / Phase 2）

> **重要变更（评审 1 严重问题 2）：** 原 4-6 周 MVP 范围过大（含 LinkedIn 调研、4 邮箱 OAuth、研究驱动草稿等），任何有过 LLM 应用上线经验的团队都知道不可能在 4-6 周交付。**重新拆分为 Phase 0（骨架）+ Phase 1（MVP）+ Phase 2（增强）**，让客户在第 3 周就能看到能用的东西，谈判节奏更稳。

### Phase 0：骨架（第 1-2 周）— 验证技术通路

目标：以最小代价跑通端到端链路，让客户尽早看到 demo。

| 模块 | 交付物 | 验证目标 |
|-----|-------|---------|
| 邮箱接入（1 个） | Marisa 的 Gmail 通过 OAuth 拉取邮件 | 确认邮件接入技术通路 |
| 字段提取 | Gemini Flash 提取 [业务规格.md §3.1](业务规格.md) 关键字段（姓名、签证类型、邮箱、来源） | 验证解析准确率 ≥85% |
| 评分 v0 | 仅 GD vs Not-GD 二分类 + 置信度（暂不做完整 4 级） | 验证评分方向正确 |
| DNQ 前置规则 | Python 函数实现 6 条 DNQ 硬规则 + 2 条软风险标记（[业务规格.md §10.2](业务规格.md)）；缺失/冲突事实转人工，不自动 DNQ | 在独立 DNQ fixture 上规则命中准确率 ≥95%，并覆盖非 DNQ 与 unknown 防误杀样本 |
| 人工审核队列 v0 | 最简列表页：Marisa 可点击「确认/修改」 | 确认人机协作流程 |
| Sheets 写入（单向） | 解析结果写入 OneSheet 指定 tab | 验证数据回流 |

**Phase 0 不做：** 多邮箱、完整 4 级评分、调研模块、回复起草、WhatsApp。

### Phase 1：MVP（第 3-6 周）— 客户可日常使用

| 模块 | 交付物 | 依赖 |

## 8. 产品建议与设计观点

### 8.1 不要过度自动化拒绝
即便 DNQ 逻辑清晰，Xpatweb 的规程要求拒绝类事项与 Marisa 及时讨论和确认。所有命中 DNQ 的 BD 线索必须保留人工介入——系统的职责是让这个步骤对 Marisa / 指定审核人 **更快**，而不是绕过人工确认。在审核队列里建一个一键"确认并发送"功能。

### 8.2 评分解释不可省略
顾问不会信任黑箱评分。每张线索卡片必须用白话展示评为 GD/MD/Bad 的**原因**。这也能加快四眼审核速度——审核人可以快速发现 AI 的误判。

### 8.3 【建议】先用 Google Sheets 作为 CRM
**这是产品建议，非既定需求。** 原始 requirement 中 CRM 选型待定（HubSpot / Salesforce / Sheets 均在候选）。鉴于 Xpatweb 现有的"单一 Tracking Sheet"文化，建议 MVP 阶段将 Google Sheets 回写（通过 Sheets API）作为初始 CRM，阻力最小、集成工作量为零，等团队信任系统后再迁移到 HubSpot。**最终 CRM 选型由客户在 [业务规格.md §11 OQ 第 2 条](业务规格.md)（P0）确认。**

### 8.4 线索来源标签价值极高——必须保留
活动代码（XP349、RSA024、ML393）代表真实的营销投入。系统必须忠实记录 `lead_source` 并在分析中暴露这一维度。这让 Xpatweb 能看到哪些活动带来 Gold 线索、哪些带来 Bad 线索——是直接的 ROI 对话素材。

### 8.5 四眼审核队列应该是 Web UI，不是邮件
如果用邮件传递审核请求，等于重建他们正在逃离的混乱。一个轻量级 Web 仪表盘（哪怕只是简单的 Next.js 应用）配上线索卡片 + 草稿编辑器 + 批准按钮，才是正确的 UX。需要适配移动端——顾问可能在手机上审核。

### 8.6 语气一致性比速度更重要
Xpatweb 的模板风格正式、精炼，使用南非英语。任何听起来"美式口语"或带有缩写的 AI 草稿都会损害品牌形象。要么仔细调整 prompt 并附上真实案例，要么增加风格校验环节。建议在 system prompt 中提供3-5封真实发出的邮件作为 few-shot 示例。

### 8.7 15天质检是独立的工作流
Sales Protocol 中的"提交前10天 Marisa 质检"**不属于**线索分级——它是案例管理功能。MVP 不要混淆这两个流程。标注为 Phase 2 的案例管理功能。

### 8.8 置信度评分防止静默失败
当提取字段存在歧义（如签证类型不明确、未提及国籍）时，系统应标记线索等待人工确认，而不是猜测。线索卡片上显示"低置信度"徽章，远好于一个自信的错误评分。

### 8.9 60秒致电规则是最核心的操作约束
这是 GD/MF/MD 线索的不可妥协 SLA。系统对这类线索最重要的事是**立即**将致电提醒送到 Leads 团队成员面前——而不是2小时后发出一封精美邮件。通知设计要追求速度：push notification 或 Slack DM，不是邮件。邮件草稿是次要的。

### 8.10 WhatsApp 发布步骤造成工作流依赖
目前线索在内部 WhatsApp 群发帖并获得点赞前，不能向外发送回复邮件。这是系统必须尊重的人工检查节点——不能在获批前自动发邮件。审核队列应清晰呈现这个门控环节。建议 AI 起草 WhatsApp 发帖文案（Box – Quality – Action 格式）来加速这一步。

### 8.11 BD 线索就是二等公民——这是设计意图
BD 线索只进内部 WhatsApp 群，不进营销线索群，团队也不致电。系统从一开始就应区别对待 BD 线索：无致电提醒、无咨询预约、精简回复。不要在 BD 路径上过度设计。

### 8.12 评分标准是地板，不是天花板
Leads 团队会根据 LinkedIn 调研和电话沟通结果覆盖 AI 评分。AI 评分应标注为"初始评估"，OneSheet 中应同时维护 `enquiry_lead_quality`（AI 评估）和 `current_quality`（互动后人工更新）——与客户现有 Tracker 完全对应。

### 8.13 调研异步执行——绝不能因此阻塞致电提醒
60秒致电规则与调研在不同的时间轴上运行。致电提醒在评分完成（线索到达约1-2分钟后）就立即触发。调研需要30-90秒，简报应在 Leads 团队成员打完第一个电话前出现在线索卡片上。UI 设计应在致电提醒旁显示"调研进行中…"的 loading 状态——绝不能等调研结束再触发提醒。

### 8.14 研究驱动草稿是第二版草稿，不是替代品
第一版草稿（模板快速填充）立即生成，不被阻塞。调研完成后生成第二版增强草稿，以"个性化版本"的形式并排展示。让 Leads 团队选择发哪个。这样4小时 SLA 永远不会因为调研任务变慢而受影响。

### 8.15 外发邮件中不能提及做过调研
草稿的语气应像团队本来就了解这些背景——而不是像机器人查过某人的资料。"I noticed you work at..." 或 "Based on your LinkedIn..." 这类表述会损害信任感。调研结果应无声地融入语气和个性化，而不是显性引用。
