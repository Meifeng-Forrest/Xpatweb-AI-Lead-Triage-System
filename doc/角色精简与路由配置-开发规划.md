## 1. 背景与目标

### 1.1 现状问题（为什么要改）

当前角色有 **6 个**：`admin` / `lead_agent` / `team_lead` / `quality_lead` / `escalation_handler` / `visa_verifier`，定义在 [auth.py:28](../backend/app/services/auth.py#L28) 的 `ROLE_PERMISSIONS` 常量。

但从权限矩阵看，**6 个角色实际只有 4 套权限**，存在两对完全重复：

| 角色 | 权限集合 | 是否重复 |
| --- | --- | --- |
| `admin` | 全部 7 项 | — |
| `lead_agent` | view + draft.edit + reject | ⟵ 与 team_lead 相同 |
| `team_lead` | view + draft.edit + reject | ⟵ 与 lead_agent 相同 |
| `quality_lead` | view + reject.confirm | — |
| `escalation_handler` | view | ⟵ 与 visa_verifier 相同 |
| `visa_verifier` | view | ⟵ 与 escalation_handler 相同 |

**根因**：角色这一个字段被迫身兼两职：

```
        ┌──────────────────────┐
角色 ──┤ ① 权限：能点哪些按钮     │
        │ ② 路由信箱：收哪类线索   │
        └──────────────────────┘
```

`escalation_handler`、`visa_verifier` 在权限上是冗余的（都只是只读），它们存在的唯一意义是充当 [routing.py](../backend/app/services/routing.py) 里 `list_routing_targets(role)` 的「信箱标签」。这导致：

- 用户管理界面角色 chip 过多，管理员分不清 `lead_agent` 与 `team_lead` 的区别（UX 复杂）；
- 想新增一类「收件人分组」就被迫造一个新角色，角色数量只增不减。

### 1.2 目标

1. **角色回归纯权限**，从 6 个收敛到 **4 个**：`admin` / `agent` / `quality_lead` / `reviewer`。
2. **路由收件人单独配置（方案 A）**：新增 `routing_rules` 配置表，把「哪类线索通知谁」从角色解耦出来，由管理员在界面上按场景勾选收件人。
3. **向后兼容**：现有 6 角色用户数据平滑迁移到 4 角色；带兜底，迁移期间通知不丢。

### 1.3 非目标（本轮不做）

- 不做「权限矩阵自助编辑 UI」——`ROLE_PERMISSIONS` 继续保留为后端常量（沿用 `TODO §2.9` 的延后决策）。
- 不接真实 Slack / Push / Email 通知通道——`Notifier` 仍为 `ConsoleNotifier`（沿用 `TODO §2.5`）。本轮只解决「收件人是谁」的配置问题，不解决「用什么渠道发」。

## 2. 设计决策

### 2.1 新角色矩阵（4 个，纯权限）

| 新角色 | 权限 | 职责（人话） | 由哪些旧角色合并 |
| --- | --- | --- | --- |
| `admin` | 全部 7 项 | 超级管理员：审批外发、配路由、管用户 | admin |
| `agent` | `lead.view` + `lead.draft.edit` + `lead.reject` | 一线顾问：看线索、改草稿、退回 | lead_agent + team_lead |
| `quality_lead` | `lead.view` + `lead.reject.confirm` | 质量负责人：确认 DNQ/Bad 拒绝 | quality_lead |
| `reviewer` | `lead.view` | 只读审阅者：升级处理 / 签证核查接收人 | escalation_handler + visa_verifier |

> 权限项不变，仍是 7 项：`lead.view`、`lead.draft.edit`、`lead.approve`、`lead.reject`、`lead.reject.confirm`、`routing.config`、`user.manage`。

**关键转变**：原本靠 `escalation_handler` / `visa_verifier` 两个角色区分「升级处理人」和「签证核查人」，合并成 `reviewer` 后，**这个区分转移到 `routing_rules` 表按场景配人**（见 2.3）。角色只回答「能做什么」，路由配置回答「谁收哪类信」。

### 2.2 路由场景（category）

路由场景沿用 [routing.py:52](../backend/app/services/routing.py#L52) `_target_role` 现有的判定分支，固化为 4 个 category：

| category | 触发条件（现有逻辑） | 含义 |
| --- | --- | --- |
| `escalation` | `lead.escalation_flag` | 升级线索 |
| `dnq_reject` | `lead.dnq_reason` | DNQ / 需拒绝确认 |
| `visa_verification` | `_needs_visa_verification(lead)` | 签证核查 |
| `standard_review` | `lead_score ∈ {GD, MF, MD, BD}` | 常规待审核 |

> category 是一个**枚举常量**（后端代码内定义），不放数据库——它是业务规则，不是配置项。可配的是「每个 category 通知谁」。

### 2.3 路由配置模型（方案 A 核心）

新增 `routing_rules` 表，存「场景 → 收件人」多对多映射，收件人**直接指到 user_id**（精确到人，最灵活）：

```sql
CREATE TABLE IF NOT EXISTS routing_rules (
    category   TEXT NOT NULL,                                    -- escalation / dnq_reject / visa_verification / standard_review
    user_id    TEXT NOT NULL REFERENCES users(user_id) ON DELETE CASCADE,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    PRIMARY KEY (category, user_id)
);
CREATE INDEX IF NOT EXISTS idx_routing_rules_category ON routing_rules (category);
```

**兜底规则（必须实现，防止漏发）**：某 category 在 `routing_rules` 中没有任何在职收件人时，**回退到所有在职 `admin`**。这样在配置表为空（刚上线 / 迁移期）时，行为与现状（全部路由给 admin）完全一致，零回归风险。

**查询收件人新逻辑**（替换现有 `list_routing_targets(role)`）：

```
1. 按 lead 算出 category
   ↓
2. 查 routing_rules 中该 category 的在职 user_id
   ├─ 有 → 用这批人
   └─ 空 → 兜底：所有在职 admin
   ↓
3. 逐个调用 Notifier 插座发通知
```

## 3. 数据库变更

### 3.1 新增表

- `routing_rules`（见 2.3 DDL）写入 [database.py](../backend/app/database.py) 的 `SCHEMA_SQL`，并在结尾日志的 `tables` 列表追加 `routing_rules`。

### 3.2 角色数据迁移（user_roles 表）

现有用户的角色需从旧值改写为新值。映射：

| 旧角色 | 新角色 |
| --- | --- |
| admin | admin |
| lead_agent | agent |
| team_lead | agent |
| quality_lead | quality_lead |
| escalation_handler | reviewer |
| visa_verifier | reviewer |

⚠️ **主键冲突风险**：`user_roles` 主键是 `(user_id, role)`。若某用户**同时**拥有 `lead_agent` 和 `team_lead`（两者都映射到 `agent`），直接 UPDATE 会撞主键。迁移 SQL 必须用「先插新、ON CONFLICT DO NOTHING、再删旧」的幂等写法：

```sql
-- 幂等迁移：先写入合并后的新角色（撞主键则跳过），再清掉旧角色
INSERT INTO user_roles (user_id, role)
    SELECT user_id, 'agent' FROM user_roles WHERE role IN ('lead_agent', 'team_lead')
    ON CONFLICT DO NOTHING;
INSERT INTO user_roles (user_id, role)
    SELECT user_id, 'reviewer' FROM user_roles WHERE role IN ('escalation_handler', 'visa_verifier')
    ON CONFLICT DO NOTHING;
DELETE FROM user_roles WHERE role IN ('lead_agent', 'team_lead', 'escalation_handler', 'visa_verifier');
```

迁移执行位置建议：放在应用启动 schema 初始化之后，做一段**幂等**的 one-off 迁移（重复执行无副作用，因为旧角色删完后 INSERT 的 SELECT 命中 0 行）。

## 8. 验收标准（Given/When/Then）

1. **角色收敛**
   - Given 系统已迁移，When 管理员打开 Users 页，Then 角色只显示 `admin` / `agent` / `quality_lead` / `reviewer` 四个。
2. **迁移无损**
   - Given 迁移前 `melissa` 拥有 `team_lead`+`lead_agent`，When 迁移完成，Then 她拥有单个 `agent` 角色，权限为 view+draft.edit+reject，且无主键冲突报错。
3. **路由兜底**
   - Given `routing_rules` 表为空，When 一条 GD 线索起草完成，Then 通知发给所有在职 `admin`（与现状一致）。
4. **路由可配置**
   - Given 管理员把 `visa_verification` 收件人配为 Willem，When 一条命中签证核查的线索起草完成，Then 通知只发给 Willem，不发给 admin。
5. **权限守护**
   - Given 一个 `agent` 用户，When 调用 `PUT /api/v1/routing/rules/{category}`，Then 返回 403（无 `routing.config`）。
6. **回归**
   - 容器后端单测全绿；`npm run lint` / `npm run build` 通过。

## 9. 风险与回滚

| 风险 | 缓解 |
| --- | --- |
| 角色迁移主键冲突（多角色用户） | 用「INSERT ON CONFLICT DO NOTHING + DELETE 旧角色」幂等写法（见 3.2） |
| 迁移期通知漏发 | `routing_rules` 空时兜底全 admin，行为等同现状，零回归 |
| 前端别处硬编码 6 角色 | 阶段三全局搜 `lead_agent|team_lead|escalation_handler|visa_verifier` 清零 |
| 旧测试断言失败 | 阶段一同步改测试，作为迁移正确性的 gate |
| `routing_rules` 指向已停用用户 | 查询收件人时 `JOIN users WHERE is_active = TRUE` 过滤 |

**回滚**：本方案对 `routing_rules` 为增量表，角色迁移为数据变更。若需回滚，可保留旧 `ROLE_PERMISSIONS` 分支 + 不删 `user_roles` 旧值（迁移前先备份 `user_roles`）。建议迁移前 `pg_dump user_roles` 留底。
