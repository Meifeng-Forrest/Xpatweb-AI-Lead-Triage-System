> 创建日期：2026-06-05

>
> 本文件取代上一版「6→4 角色 + 路由独立页」方案。现状代码已落地「4 角色 + 独立 Routing 页」，本轮在此基础上继续演进。

## 1. 背景与目标

### 1.1 现状（已实现，作为本轮起点）

- **角色 4 个**（纯权限，定义在 [auth.py:28](../backend/app/services/auth.py#L28) `ROLE_PERMISSIONS`）：
  - `admin`（全部 7 项）、`agent`（看+改草稿+退回）、`quality_lead`（看+确认退回）、`reviewer`（只看）。
- **路由独立配置**：`routing_rules` 表（category → user_id）+ 独立 Routing 页（[App.tsx:1691](../src/App.tsx#L1691) `RoutingRulesView`），按场景勾收件人。

### 1.2 现状的两个问题（为什么还要改）

**问题 A：`admin` 一个角色身兼两类完全不同的职责，过载。**

```
admin ──┬── 系统管理：管账号(user.manage) + 配路由(routing.config)   ← 高危，应极少数人
        └── 业务审批：通过外发(lead.approve)                       ← 日常，应多人分担
```

- 「能改别人账号、改路由」是高危操作，却和天天要干的「审批线索外发」绑在同一个角色上。
- **副作用 = 一个潜在死锁**：外发强制四眼（[leads.py:773](../backend/app/api/leads.py#L773)「通过的人 ≠ 改草稿的人」）。当前只有 `admin` 能 `lead.approve`，若这唯一的 admin 自己改了草稿，这条线索就**没人能批**——他被四眼挡住，别人又没 approve 权限，卡死。

**问题 B：路由是独立一页，跟「这个人是谁、能干啥」割裂。**

- 管理员要在 Users 页配角色、再切到 Routing 页配收件，两处来回跳。
- Routing 页把整份用户名单按 4 个 category 各画一遍（[App.tsx:1691+](../src/App.tsx#L1691)），重复严重、看不到「某个人收哪几类」的反向视图。

### 1.3 目标

1. **角色按职责重切为 5 个**：`superadmin` / `approver` / `agent` / `quality_lead` / `reviewer`。把「管系统」与「审批业务」拆开，并让审批多人化（顺带解死锁）。
2. **路由并入用户表**：删除独立 Routing 页，把「这个人收哪几类线索」做成用户行里的多选（category 复选框），一页搞定「谁、能干啥、收什么信」。
3. **向后兼容**：现有用户、现有 `routing_rules` 数据平滑迁移，迁移期通知不丢。

### 1.4 非目标（本轮不做）

- 不做「权限矩阵自助编辑 UI」——`ROLE_PERMISSIONS` 继续是后端常量。
- 不接真实 Slack / Push / Email——`Notifier` 仍为 `ConsoleNotifier`，只解决「收件人是谁」。
- 不改 `user_roles` 表结构——仍是多对多表；本轮只在 UI + 后端校验上约束「一人一角色」（见 §2.4），为将来若需放开多角色留门（零迁移）。

## 2. 设计决策

### 2.1 新角色矩阵（5 个，按职责重切）

权限项不变，仍是 7 项：`lead.view`、`lead.draft.edit`、`lead.approve`、`lead.reject`、`lead.reject.confirm`、`routing.config`、`user.manage`。

| 角色 | view | draft.edit | approve | reject | reject.confirm | routing.config | user.manage | 职责（人话） | 人数 |
| --- | :--: | :--: | :--: | :--: | :--: | :--: | :--: | --- | --- |
| **superadmin** | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | 超级管理员：管账号、配路由，并兜底全部业务权限 | 极少（1–2） |
| **approver** | ✅ | ❌ | ✅ | ❌ | ❌ | ❌ | ❌ | 审批人：通过外发，**可多人分担** | 多人 |
| **agent** | ✅ | ✅ | ❌ | ✅ | ❌ | ❌ | ❌ | 一线顾问：改草稿、退回 | 多人 |
| **quality_lead** | ✅ | ❌ | ❌ | ❌ | ✅ | ❌ | ❌ | 质量负责人：确认退回（复核拒绝） | 少数 |
| **reviewer** | ✅ | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ | 只读接收人：升级 / 签证核查，线下处理 | 按需 |

**和现状（4 角色）的差异：**

```
admin  ──拆──>  superadmin（系统管理 + 兜底）
                  +
                approver（业务审批，新角色，可多人）   ← 解决 §1.2 的过载与死锁
agent / quality_lead / reviewer  原样保留
```

**关键设计点 —— 两道职责分离（separation of duties）必须保住：**

1. **外发线**：`agent` 改草稿 → `approver`/`superadmin` 通过（且代码强制「通过者 ≠ 改草稿者」）。
   - 把 `approve` 放开给多个 `approver`，等于给四眼规则**备好第二双眼睛**，§1.2 的死锁自然解除。
2. **拒绝线**：`agent` 退回 → `quality_lead` 确认退回。
   - 因此 `agent` 和 `quality_lead` **不能合并**，否则「退回」和「确认退回」变同一个人，复核失效。

> 这也是上一版讨论「能否压成 3 角色」的最终结论：**角色好坏看「每个角色 = 一份说得清的工作」，不看数量**。本轮该拆的拆（admin）、该保的保（agent/quality_lead 分离）。

### 2.2 路由模型（不变）+ 配置入口（改）

- **路由场景 category 不变**，仍是 4 个枚举（[routing.py:11](../backend/app/services/routing.py#L11) `ROUTING_CATEGORIES`）：

  | category | 触发条件 | 含义 |
  | --- | --- | --- |
  | `escalation` | `lead.escalation_flag` | 升级线索 |
  | `dnq_reject` | `lead.dnq_reason` | DNQ / 需拒绝确认 |
  | `visa_verification` | `_needs_visa_verification(lead)` | 签证核查 |
  | `standard_review` | `lead_score ∈ {GD, MF, MD, BD}` | 常规待审核 |

- **`routing_rules` 表结构不变**（`category, user_id` 多对多，[database.py:156](../backend/app/database.py#L156)）。
- **路由引擎不变**：`route_after_draft` 仍按 category 查 `list_recipients(category)`（[routing.py:19](../backend/app/services/routing.py#L19)）。
- **改的只是「配置方向」**：从「按 category 选人」（独立页）→「按人选 category」（用户表内多选）。底层是同一张表的**转置**，所以引擎零改动。

### 2.3 兜底角色：`admin` → `superadmin`（⚠️ 必改，否则漏发）

现有兜底逻辑（[users.py:257-260](../backend/app/repositories/users.py#L257)）：某 category 无在职收件人时，回退到所有在职 `admin`：

```python
# users.py:260
return await UserRepository(self.pool).list_routing_targets("admin")
```

角色迁移后 `admin` 不复存在，此处若不改将返回空 → **通知漏发**。必须改为兜底 `superadmin`：

```python
return await UserRepository(self.pool).list_routing_targets("superadmin")
```

同步：[api/routing.py](../backend/app/api/routing.py) 中 `fallback_to_admin` 文案/字段语义改为「兜底 superadmin」（字段名可保留或改名 `fallback_to_superadmin`，二选一，注意前端同步）。

### 2.4 角色单选（一人一角色）

本轮把角色从「多选」收敛为「**单选**」，一个用户只挂一个角色。原因：

1. **多选的历史理由已消失**——以前要靠 `agent + reviewer` 这种组合，让一个人「既有权限、又当路由信箱标签」。本轮路由已拆成 per-user 独立轴（§2.2），「收哪类信」由路由场景多选决定，跟角色无关，多选不再需要。
2. **天然守住职责分离**——单选下一个人不可能同时勾 `agent`（退回）和 `quality_lead`（确认退回），拒绝线复核不会被误配破坏。
3. **多选的其余组合要么冗余要么危险**：`superadmin + 任意` 冗余（已含全部）、`reviewer + 任意` 冗余（view 是子集）、`agent + quality_lead` 危险（破坏复核）。真正有意义的「资深双岗」极少见，需要时用 `superadmin` 或将来定义组合角色表达，**不靠临时勾两个框**。

**实现约束（不改表，留门）：**

- `user_roles` 表结构**不变**（仍多对多），仅在**后端校验**「roles 长度必须 = 1」+ **前端 radio** 落地单选。将来若要放开多角色，去掉校验即可，零迁移。
- 权限 = 该单一角色的权限集合（`permissions_for_roles` 逻辑不变，对单元素列表照常工作）。

## 3. 数据库变更

### 3.1 表结构

- **无新增表**。`routing_rules`、`user_roles` 结构均不变。

### 3.2 角色数据迁移（幂等）

在 [database.py](../backend/app/database.py) 现有迁移段（line 167-173，把 6→4 的那段）之后，**追加** 5 角色迁移。映射：

| 现角色 | 新角色 |
| --- | --- |
| admin | superadmin |
| agent | agent（不变） |
| quality_lead | quality_lead（不变） |
| reviewer | reviewer（不变） |

幂等 SQL（先插新、撞主键跳过、再删旧；重复执行无副作用）：

```sql
-- admin → superadmin（保留其全部权限，仅改名）
INSERT INTO user_roles (user_id, role)
    SELECT user_id, 'superadmin' FROM user_roles WHERE role = 'admin'
    ON CONFLICT DO NOTHING;
DELETE FROM user_roles WHERE role = 'admin';
```

> `approver` 是**新增能力**，不做自动迁移——上线后由 superadmin 在 Users 页手动给需要审批权的人勾上 `approver`。迁移期内，原 admin 已变 superadmin 仍能 approve，业务不中断。

### 3.3 seed 用户角色

[auth.py](../backend/app/services/auth.py) 的 seed 用户（约 line 213-217）：原 `admin` seed 改为 `superadmin`；可按需给某个 seed 用户加 `approver` 做演示数据。

## 5. 前端改动清单（界面更改重点）

### 5.1 目标界面 —— 表格只读总览 + 弹窗编辑

删除独立 Routing 页。Users 表**回归只读总览**：每行只展示「角色标签 + 路由标签」，状态做成头像旁的小标识，编辑动作收进**弹窗**。这样表格信息密度低、扫读快；改动集中在 Modal 里完成。

**① 列表（只读，不出现任何输入控件）：**

```
┌──────────────────────────────────────────────────────────────────────────────────────┐
│ User Management                                                            [+ Add User] │
│ ───────────────────────────────────────────────────────────────────────────────────── │
│  USER                          ROLES                    ROUTING (收哪类线索)            │
│ ───────────────────────────────────────────────────────────────────────────────────── │
│  🟢Ⓜ Melissa  melissa@...      [Agent]                  [升级] [常规]            [编辑]  │
│  🟢Ⓜ Marisa   marisa@...       [Quality Lead]           [DNQ]                    [编辑]  │
│  🟢Ⓦ Willem   willem@...       [Reviewer]               [签证]                   [编辑]  │

└──────────────────────────────────────────────────────────────────────────────────────┘
```

- **STATUS 不再单列**：用头像左上角的小圆点表示——🟢 在职 / ⚪ 停用（hover 显示 `Active` / `Disabled`）。
- **PASSWORD 不再单列**：重置密码挪进编辑弹窗。
- **ROLES 列**：只读角色 chip（管「能干什么」）。
- **ROUTING 列**：只读路由标签（管「收什么信」），无显式配置时显示「—（兜底接收全部）」灰字。
- 点行尾 **[编辑]**（或整行点击）→ 打开编辑弹窗。

**② 编辑弹窗（所有改动集中在这里）：**

```
┌─────────────────────────────────────────────┐
│  编辑用户 · Jerry                        [✕]  │
│ ─────────────────────────────────────────── │
│  显示名   [ Jerry                          ] │

│                                              │
│  状态     ◉ 在职   ○ 停用                     │

│                                              │

│   [✓]升级  [ ]DNQ  [ ]签证  [ ]常规            │
│   ⚠ 全不勾 → 该类由 superadmin 兜底            │
│                                              │
│  重置密码  [ 新密码………… ]  [重置]            │
│ ─────────────────────────────────────────── │
│                        [取消]  [保存]         │
└─────────────────────────────────────────────┘
```

**两个独立的轴，弹窗里也分两块、不混淆：**

- **角色区 = 单选 radio（管「能干什么」）**：5 个角色，一人一角色（见 §2.4）。
- **路由场景区 = 多选（管「收什么信」）**：4 个场景。

**交互细节：**

1. **路由场景区仅 `routing.config`（superadmin）可编辑**；非 superadmin 打开弹窗时，该区**只读置灰**（仍可看到该用户收哪几类，但不能改）。角色区、状态、密码按各自权限（`user.manage`）控制。
2. 保存时：角色变更走现有 `PUT /api/v1/users/{id}`；路由变更走新 `PUT /api/v1/users/{id}/routing-categories`。两者均补脱敏日志；保存成功后刷新该行只读展示。
3. 某 category 全员未勾 → 沿用兜底（superadmin 收），弹窗内用一行小字提示「全不勾 → superadmin 兜底」。
4. **停用用户**不参与路由（兜底查询已 `is_active = TRUE` 过滤）；列表头像显示 ⚪。
5. **去重账号**：`admin@example` / `admin@local`、`Jerry×2` 这类 seed 重复，邮箱用 `@local` 灰字区分，别让两条名字一样的并排难辨。

## 8. 验收标准（Given/When/Then）

1. **角色重切**：Given 已迁移，When 打开 Users 页，Then 角色可选 `superadmin` / `approver` / `agent` / `quality_lead` / `reviewer` 五个，无 `admin`。
1b. **角色单选**：Given 编辑用户弹窗，When 角色区为 radio，Then 只能选 1 个；提交 2 个角色的请求被后端 422 拒绝。
2. **迁移无损**：Given 迁移前用户拥有 `admin`，When 迁移完成，Then 其角色为 `superadmin`、7 项权限齐全、无主键冲突。
3. **审批多人化 + 死锁解除**：Given 用户 A（approver）改了草稿，When A 自己点通过，Then 409（四眼）；When 另一 approver B 点通过，Then 成功外发。
4. **路由并入用户表**：Given superadmin 在 Users 行给 Willem 勾「签证」，When 命中签证核查的线索起草完成，Then 仅通知 Willem。
5. **路由兜底**：Given 某 category 无人勾选，When 该类线索起草完成，Then 通知所有在职 `superadmin`。
6. **权限守护**：Given 一个 `agent`，When 调 `PUT /users/{id}/routing-categories`，Then 403。
7. **回归**：容器后端单测全绿；`npm run lint` / `npm run build` 通过。

## 9. 风险与回滚

| 风险 | 缓解 |
| --- | --- |
| 兜底角色没改，迁移后漏发通知 | §2.3 强制把 `list_routing_targets("admin")` 改 `superadmin`，迁移测试覆盖兜底收件 |
| 两套路由写路径（per-category + per-user）不一致 | 建议只留 per-user 写入；保留 per-category 仅做只读或删除 |
| `approver` 无人配，导致没人能审批 | 迁移后原 admin = superadmin 仍能 approve，业务不中断；上线提示 superadmin 尽快指派 approver |
| 前端别处硬编码 `admin` 角色 / 旧 Routing 页 | 阶段四全局搜 `admin`（角色语境）、`RoutingRulesView`、`'routing'` 清零 |
| 角色迁移主键冲突 | INSERT ON CONFLICT DO NOTHING + DELETE 旧角色（§3.2 幂等） |

**回滚**：角色迁移前 `pg_dump user_roles` 留底；`routing_rules` 数据本身不动（仅配置入口换方向），可安全回退到独立 Routing 页。
