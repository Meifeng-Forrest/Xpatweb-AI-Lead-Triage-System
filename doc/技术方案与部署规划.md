## 1. 技术栈决策

**部署方式：Docker 容器化，部署至客户服务器。**

| 层级 | 选型 | 说明 |
|------|------|------|
| 后端框架 | Python + FastAPI | 异步支持好，便于统一托管 Gemini/Graph/数据库调用 |
| 数据库 | PostgreSQL（自建，Docker Compose 一并打包） | 不用 Supabase，避免厂商锁定，迁移友好 |
| 消息队列 | Redis + Celery 或 Redis Streams | 调研任务异步执行（不阻塞 60 秒致电提醒） |
| AI 提取 | Gemini Flash（temperature=0） | 业务规格 §10.1 |
| AI 评分/起草 | Gemini | 业务规格 §10.3 + §10.5 |
| DNQ 规则 | Python 函数（零 token，确定性、可测试） | 只执行明确结构化事实；缺失/冲突字段转人工，见业务规格 §4.1 + §10.2 |
| 容器化 | Docker Compose（app + postgres + redis） | 一键拉起，环境变量 `.env` 注入 |
| 前端（审核队列） | Next.js 或轻量 React（移动端适配） | 四眼审核 UI |
| 邮件接入 | Microsoft Graph API + OAuth2 | 业务规格 §11 OQ 第 1 条已初步确认 Outlook |

> **注意：** 当前决策为所有 LLM 阶段统一走后端 Gemini 服务层；前端不再直接持有或调用 Gemini Key，详见 `TODO.md §1.1`。
>
> **DNQ 分工原则：** Gemini 负责从自然语言中提取事实，数据校验层负责识别缺失与冲突，Python 仅执行明确的硬规则。Python 不会自行提高上游事实的准确性，因此 unknown/低置信度场景不得自动 DNQ，必须进入人工复核。

## 2. 邮件接入（Microsoft Graph）

### 2.1 应用注册信息（App Registration: AI Lead Triage）

| 字段 | 存放位置 |
|------|---------|
| Application ID | `.env` → `MS_CLIENT_ID` |
| Tenant ID | `.env` → `MS_TENANT_ID` |
| Secret ID | `.env` → `MS_SECRET_ID` |
| Client Secret | `.env` → `MS_CLIENT_SECRET` |

> 🔐 **安全约定：** 应用注册的所有标识符与密钥（Application ID / Tenant ID / Secret ID / Client Secret）完整值**仅**存在于根目录 `.env`（已加入 `.gitignore`），本文档及任何入库文件**不记录真实值**。轮换密钥时更新 `.env` 即可，无需改动本文档。

### 2.2 API 权限

| 权限 | 类型 | 说明 | 状态 |
|------|------|------|------|
| `Mail.Read` | Application | 读取所有邮箱邮件 | ✅ Granted |
| `User.Read` | Delegated | 登录并读取用户资料 | ✅ Granted |

`Mail.Read` 是 **Application 级别**，无需每个用户单独授权，可直接读取四个品牌收件箱。

### 2.3 覆盖范围

- 平台：Microsoft 365 / Outlook
- 品牌收件箱：XP、RISA、VLS、SMV 全部使用 Outlook

### 2.4 接入方式

采用 **OAuth2 Client Credentials Flow**（Application 级别权限，无需用户登录）：

```http
POST https://login.microsoftonline.com/{MS_TENANT_ID}/oauth2/v2.0/token

client_id     = {MS_CLIENT_ID}
client_secret = {MS_CLIENT_SECRET}
scope         = https://graph.microsoft.com/.default
grant_type    = client_credentials
```

获取 token 后调用：

```http
GET https://graph.microsoft.com/v1.0/users/{mailbox}/messages
```

四个品牌收件箱各轮询一次，按来源打上对应的 `source_box` 标签（XP / RISA / VLS / SMV）。

## 3. 容器拓扑（Docker Compose）

```
┌──────────────────────────────────────────────┐
│                客户服务器                       │
│                                                │
│  ┌──────────┐  ┌──────────┐  ┌─────────────┐  │
│  │   app    │  │ postgres │  │    redis    │  │
│  │ FastAPI  │─▶│  数据持久 │  │ 队列 + 缓存  │  │
│  │ + Celery │─▶│   化      │  │             │  │
│  └────┬─────┘  └──────────┘  └─────────────┘  │
│       │ .env 注入所有密钥与连接串                │
│       ▼                                        │
│  ┌──────────┐                                  │
│  │ frontend │  四眼审核 UI（Next.js / React）   │
│  └──────────┘                                  │
└──────────────────────────────────────────────┘
        │ 出站
        ├──▶ Google Gemini API
        └──▶ Microsoft Graph API（邮件拉取）
```

- 所有服务由一份 `docker-compose.yml` 编排，`.env` 统一注入。
- `postgres-data/`、`redis-data/` 为本地持久化卷，已在 `.gitignore` 中忽略。

## 4. 环境变量清单

完整值仅保存在根目录 `.env`（不入库）。仓库不保留 `.env.example`，避免客户服务器部署资料和内部字段模板误提交。关键项：

| 变量 | 用途 |
|------|------|
| `LLM_PROVIDER` / `LLM_API_PROTOCOL` / `LLM_BASE_URL` | 当前 LLM 服务商、协议与接口地址（当前以 Gemini 后端服务层为准） |
| `GEMINI_API_KEY` / `GEMINI_BASE_URL` | 后端调用 Gemini 的鉴权和接口地址 |
| `GEMINI_MODEL_EXTRACT` / `GEMINI_MODEL_SCORE` / `GEMINI_MODEL_DRAFT` / `GEMINI_MODEL_RESEARCH` | 提取、评分、起草、调研阶段模型名 |
| `KIMI_API_KEY` / `KIMI_BASE_URL` / `KIMI_MODEL` | 历史兼容变量，当前不作为主链路使用 |
| `SHENGSUANYUN_API_KEY` / `SHENGSUANYUN_BASE_URL` / `SHENGSUANYUN_MODEL` | ShengSuanYun OpenAI 兼容备用路由，填写 Key 和模型后测试 |
| `MS_TENANT_ID` / `MS_CLIENT_ID` / `MS_CLIENT_SECRET` / `MS_SECRET_ID` | Microsoft Graph 邮件接入 |
| `MAILBOX_XP/RISA/VLS/SMV` | 四个品牌收件箱地址（待客户提供） |
| `DATABASE_URL` | PostgreSQL 连接串 |
| `REDIS_URL` | Redis 连接串 |

## 6. 安全注意事项

1. **密钥隔离**：所有密钥仅存 `.env`，已被 `.gitignore` 忽略；入库文件只保留脱敏摘要。
2. **本仓库历史**：本规划写入前仓库无 `.gitignore`，已补建。请确认 `.env` 从未被 `git add`（可执行 `git status` 核对 `.env` 不在追踪列表）。
3. **POPIA 合规**（PRD §5）：客户 PII 除 LLM/Graph 外不外发；部署在客户服务器，数据不出境。
4. **密钥轮换**：Microsoft Client Secret 有有效期，到期需在 Azure 门户重新生成并更新 `.env`。
