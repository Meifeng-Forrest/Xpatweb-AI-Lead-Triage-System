# AI 线索分级处理系统 — 业务规格文档
> 客户：Xpatweb（南非移民签证服务公司）
> 分析来源：Historical Leads、Lead Score Sheet（PDF）、Lead Responses (WP).docx、Sales Protocol（6个Sheet）
> 最后更新：2026年4月

---

## 1. 业务背景

Xpatweb 是一家南非移民咨询公司，旗下运营**多个品牌**，每个品牌有独立的邮箱收件箱：
- **XP** — Xpatweb（主品牌）
- **RISA** — Retire In South Africa（退休移民南非）
- **VLS** — Visa Litigation Services（签证诉讼服务）
- **SMV** — SubmitMyVisa

线索通过网页联系表单、邮件、电话和 WhatsApp 进入。服务费用从 R2,840（非正式核查）到最高约 R167,200（紧急法庭申请）不等。

**核心问题：** 约 35% 的线索是低价值或无价值的，大量浪费顾问时间。目前团队需要手动完成打分、路由、起草回复和跟踪每一条线索。系统需要将这些步骤自动化。

## 2. 线索分级体系

### 2.1 评级层级

**系统枚举（4级，与客户 Sales Protocol 完全对齐）：**

| 级别 | 标签 | 含义 |
|------|------|------|
| GD | Good / Gold | 高价值 → 立即安排咨询 |
| MF | Medium Follow Up | 值得跟进 → 确认信息 + 主动追踪 |
| MD | Medium | 边际价值 → 发报价，自动跟进 |
| BD | Bad | 低价值或不合格 → 发报价、DNQ 或拒绝邮件；仅发内部群 |

**关于 BD 与 DNQ 的关系：**
- BD 是评级，DNQ 是 BD 之下的子分类（特定的拒绝原因，见 §4）
- 所有 DNQ 线索均为 BD 评级，但 BD 不一定都是 DNQ（也可能是低价值但仍可报价）
- 系统字段：`lead_score` ∈ {GD, MF, MD, BD}；当 `lead_score = BD` 且命中 DNQ 规则时另置 `dnq_reason` 字段

**历史数据说明（n=202）：** 历史表格使用过混合标签 "Good / Bad / MD / MF / BD"，其中 "Good" ≡ GD，"Bad" ≡ BD。这是历史数据录入不一致，不代表系统存在5级体系。系统统一使用 4 级枚举。

### 2.2 个人线索评分标准（来源：Lead Score Sheet PDF）

**Gold（GD）— 签证类型 + 以下支撑条件：**
- 签证类型为以下之一：Retired Person's Visa / Financially Independent Permit / Remote Work Visa / High Net Worth
- 有显赫职称（医生、总监等）
- 第一世界国家国籍
- 高学历（NQF 8级及以上）
- 有紧迫性或申请多个签证
- 邮件措辞连贯、用词显示受过高等教育

**Mid-Field（MF）：**
- 净资产未明确
- 职称未明确
- 有紧迫性或申请多个签证
- 邮件措辞连贯、用词显示有教育背景

**Medium（MD）：**
- 净资产未明确
- 职称未明确
- 仅需单一签证协助
- 邮件措辞连贯、用词显示有教育背景

**Bad（BD）：**
- 净资产未明确
- 职称未明确
- 仅需单一签证协助
- 邮件措辞混乱、存在拼写错误

### 2.3 企业线索评分标准

**Gold（GD）：**
- 企业公司邮箱域名
- 发件人为 Director / HR / PA to Director（董事/人事/董事助理）
- 申请多种签证服务（非单纯签证核查或评估）

**Mid-Field（MF）：**
- 公司邮箱域名
- 发件人部门不明确或非高级职位
- 仅申请签证核查或评估

**Medium（MD）：**
- Gmail 个人邮箱
- 小公司或公司不明确
- 仅申请签证核查或评估

### 2.4 评分覆盖（情境判断）

**评分标准是起点，不是上限。** Leads 团队还会：
- 在 **LinkedIn** 上查询线索，核实职称和公司信息
- 在电话沟通中重新评估质量（紧迫性、必要性、支付能力）
- 根据情境因素提升评分等级

**来自操作规程的真实案例：** 一位丈夫在黎巴嫩的线索通常会评为 MF，但因为使用 Xpatweb 服务成为刚需（即便费用通常超出其预算），最终被升级为 GD——并成功转化。

**对 AI 系统的影响：** 输出评分的同时必须输出置信度。置信度低的线索应标记供人工判断，不得自动执行。

## 4. 资格规则（DNQ 硬规则 + 软规则）

### 4.1 DNQ 硬规则（命中即进入拒绝审核路径）

仅在事实判断明确、几乎不存在反例时使用。命中后系统将 `lead_score = BD` 并填入 `dnq_reason`，邮件草稿走拒绝模板路径；**仍须由 Marisa / 指定审核人确认后发送**，系统不得自动发送拒绝邮件。

**来源边界：** Lead Responses 原始 Word 中存在更多拒绝模板（例如 Points-Based System、Visitor 11(6) 关系不足）。本系统只把事实门槛清晰、可由结构化字段判定的规则放入硬规则；其余拒绝模板保留为人工审核或软风险提示。

**准确性与职责边界：**
- LLM 负责理解邮件自然语言并提取结构化事实；Python 函数只负责按照已确认事实执行规则，不能猜测缺失信息。
- 只有规则所需字段均明确、无冲突且满足触发条件时才能命中 DNQ。字段为 `null/unknown`、提取结果互相矛盾或证据不足时，必须转人工补充/复核，**不得把“不知道”当成“不符合”**。
- Python 硬规则的优势是同一输入永远得到同一结果，且每条规则可单独测试和审计；误判风险主要来自上游提取错误、规则定义错误或规则未及时更新，而不是 Python 本身。
- 上线验证必须同时覆盖：应命中的 DNQ、不得命中的非 DNQ、字段缺失/矛盾的 unknown 样本，避免只测命中率却忽略误杀。

| 规则编号 | 触发条件 | 原因 |
|---------|---------|------|
| DNQ-01 | Critical Skills Visa + 无正式工作邀请函 | 无工作邀请无法受理 |
| DNQ-02 | Permanent Residence + 当前无有效南非签证 | 核心资格要求未满足 |
| DNQ-03 | Permanent Residence（基于工作签证路径）+ 未持有符合要求的工作签证满4年 | 原始模板包含 "PR not held Work Visa for 4 years" 拒绝场景 |
| DNQ-04 | Relative's Visa / Spousal pathway + 婚姻未正式登记/传统婚姻 | DHA 要求正式登记；传统婚姻不适用 |
| DNQ-05 | Appeal（上诉）+ 已超过决定后10个工作日 | 上诉期已过；DHA 不受理逾期申请 |
| DNQ-06 | Visitor Visa 11(1) + 第一世界国家国籍 | 该类国籍免签；服务不适用 |

### 4.2 软规则（标记为高风险，转人工评估）

不进自动拒绝路径，仅设置 `risk_flags` 字段并附说明，**强制由人工 Leads 团队判定**。这些场景下原始模板存在多种例外，硬性拒绝会误伤合格线索。

| 规则编号 | 触发条件 | 备注 |
|---------|---------|------|
| RISK-01 | Points-Based System + 薪资低于 R650,976 或总分可能不足 | 原始模板有 PBS 拒绝邮件，但 PBS 是 100 分总分制；薪资只是其中一项。其他维度（学历、年龄、工作经验等）若得分高，仍可能达标。**禁止仅因薪资低于该值硬拒**，应转人工确认。 |
| RISK-02 | Visitor Visa 11(6) + 关系稳定性证据不足 | 原始模板有 11(6) 拒绝邮件，并要求 well-established relationship；婚姻初创、同居时间短、关系证据薄弱均属此类。**实际门槛由顾问按案情判断**，不可硬编码具体天数。 |

## 6. 完整线索生命周期流程

### 6.1 新线索处理流程（来源：Sales Protocol — NEW LEADS CYCLE Sheet）

```
1. 新邮件 / 联系表单进入收件箱
        │
        ▼
2. 判断线索质量（评分标准 + LinkedIn 查询）
        │
        ▼
3. 按评分执行动作：
   GD  → 60秒内致电 → 通话中预约咨询（或按偏好发邮件/WhatsApp）
   MF/MD → 60秒内致电 → 确认补充信息
   BD  → 发报价或 DNQ 邮件（不致电）
        │
        ▼
4. 将线索发布到 XP 内部 WhatsApp 群
        │
        ▼
5. 等待内部群的点赞批准
        │
        ▼
6. GD/MF/MD → 同时发布到 WhatsApp 营销线索群
   BD        → 仅发内部群，不发营销群
        │
        ▼
7. 发送邮件回复（只有获得点赞后才能发）← 4小时 SLA
        │
        ▼
8. 更新 Tracking Sheet（OneSheet）
        │
        ▼
9. 将邮件归档到对应文件夹（按品牌 + 评级）
        │
        ▼
10. 开始跟进序列（见 §6.3）
```

### 6.2 线索流水线阶段（OneSheet Status 字段）

1. First Response Sent（已发初次回复）
2. Engaging（跟进中）
3. Costing Sent（已发报价）
4. Consultation Scheduled（已安排咨询）
5. Time Proposed（已提议时间）
6. Consultation Completed（咨询已完成）
7. TOE Shared（已发服务协议）
8. Invoice Shared（已发发票）
9. First Follow Up（第一次追踪）
10. Second Follow Up（第二次追踪）
11. Nurture（持续培育）
12. Final Follow Up（最终追踪）
13. Converted（已转化）
14. Lead Closed（线索关闭）

**关闭原因：**
- Converted（已转化）
- Do Not Offer Service（不提供该服务）
- Does Not Qualify（不符合资格）
- No Longer Requires Service（不再需要服务）
- Unaffordability（价格承受不起）
- Went with Another Company（选择了其他公司）
- Completing Themselves（自行办理）
- Email Bounced（邮件退回）
- Researching Only（仅在调研）
- Missed Multiple Consultations（多次错过咨询）

### 6.3 跟进时间节点

**针对无响应 GD/MF/MD 线索（初次回复后）：**
- 第1次跟进：初次回复后1天
- 第2次跟进：第1次跟进后2天
- 最终跟进：第2次跟进后3个工作日

**GD 线索：** 每阶段均为个性化跟进
**MF/MD 线索：** 第1、2次自动跟进；最终跟进为人工发送
**注意：在 Tracker 跟进字段中填写日期会自动触发跟进邮件——GD 线索除非日期已过，否则不填日期**

**咨询后跟进（若无响应）：**
- 3个工作日后
- 2个工作日后
- 3个工作日后
- 最终跟进

### 6.4 WhatsApp 群发布格式

[BOX] – [QUALITY] – [ACTION]

### 6.5 SLA 与不可违反规则

| 规则 | 要求 |

## 10. System Prompts（两步法 + 前置规则）

> **架构原则（评审建议）：** 不再用一个大 prompt 同时做提取+评分+DNQ。改成三层处理：
>
> 1. **§10.1 字段提取**（Gemini Flash，temperature=0）— 只提取，不判断
> 2. **§10.2 DNQ 前置规则函数**（Python 代码，零 token 成本）— 6 条硬规则在 LLM 前命中即直接 BD 并进入拒绝审核路径
> 3. **§10.3 评分 Prompt**（Gemini + Few-Shot）— 仅在未命中 DNQ 时执行
> 4. **§10.4 调研 Agent**（Gemini + Web Search） — 见 §12.6
> 5. **§10.5 研究驱动回复起草**（Gemini）— 见 §12.7
>
> 这样由 LLM 负责“读懂来信并整理事实”，由 Python 负责“按明确规则执行”，人工负责“处理缺失、冲突和边界情况”。硬规则交给代码后结果确定且可测试，但系统准确性仍依赖字段提取质量、规则正确性与持续回归验证。

### 10.2 DNQ 前置规则函数（Python 伪码 — 在 LLM 评分前执行）

> **重要：** 仅 6 条硬规则（§4.1）放在此处。**PBS 薪资/总分门槛**和 **Visitor 11(6) 关系不足**已下放为软规则（§4.2），不在此函数判定。
>
> **执行护栏：** 规则函数只能在所需字段存在且取值明确时返回 DNQ。任何 `null/unknown`、字段冲突或无法从原文确认的情况，都返回非自动 DNQ 并附加人工复核标记；禁止让 LLM 或规则函数猜测否决事实。

```python
def apply_dnq_rules(extracted: dict) -> tuple[bool, str | None]:
    """
    Returns (is_dnq, dnq_reason). If is_dnq=True, skip §10.3 scoring and route to Marisa.
    These are deterministic rules — never delegated to the LLM.
    Mirrors §4.1 exactly. PBS salary and Visitor 11(6) relationship are
    SOFT RULES (see §4.2 + apply_risk_flags below) — not auto-rejected.
    """
    visa = extracted.get("visa_category")

    # DNQ-01: Critical Skills without formal job offer
    if visa == "Critical Skills Work Visa" and extracted.get("has_job_offer") is False:
        return True, "DNQ-01"

    # DNQ-02: PR application without current SA visa
    if visa in ("Permanent Residence Permit", "PR (Financially Independent)") \
       and not extracted.get("current_visa"):
        return True, "DNQ-02"

    # DNQ-03: PR via work visa route without 4 years on qualifying work visa
    if visa == "Permanent Residence Permit" \
       and extracted.get("pr_route") == "work_visa" \
       and (extracted.get("qualifying_work_visa_years") or 0) < 4:
        return True, "DNQ-03"

    # DNQ-04: Relative's / spousal pathway with unregistered/traditional marriage
    if visa in ("Relative's Visa", "Relative's Spouse Visa", "Visitors Visa Section 11(6)") \
       and extracted.get("marriage_type") in ("traditional", "unregistered"):
        return True, "DNQ-04"

    # DNQ-05: Appeal beyond 10 working days
    if visa == "Appeal Application" and is_beyond_10_working_days(extracted.get("rejection_date")):
        return True, "DNQ-05"

    # DNQ-06: Visitor 11(1) from visa-exempt (first-world) nationality
    if visa == "Visitors Visa Section 11(1)" and extracted.get("is_first_world") is True:
        return True, "DNQ-06"

    return False, None


def apply_risk_flags(extracted: dict) -> list[str]:
    """
    Returns a list of soft-risk flag codes. These do NOT auto-reject — they
    add a `risk_flags` field to the lead so the LLM scorer downgrades confidence
    and the leads team is forced to make a judgment call.
    """
    flags = []
    visa = extracted.get("visa_category")
    salary = extracted.get("annual_salary_zar")

    # RISK-01: PBS with weak salary / total-score signal (NOT a hard reject —
    # PBS is a 100-point system; salary is one factor among many)
    if visa == "Points-Based System" and (
        (salary is not None and salary < 650_976) or extracted.get("pbs_total_score_below_100") is True
    ):
        flags.append("RISK-01")

    # RISK-02: Visitor 11(6) with weak relationship evidence
    # (well-established relationship is required; no specific duration is hardcoded —
    # consultant must judge case-by-case)
    rel = extracted.get("relationship_duration")
    if visa == "Visitors Visa Section 11(6)" and rel in (
        "less_than_1_month", "newly_married", "weak_evidence", "unspecified"
    ):
        flags.append("RISK-02")

    return flags
```

> **数据校验：** §13.2 显示 Visitors Visa 11(1) 的 10 条样本全部为 Bad，与 DNQ-06 完全吻合。Chinese Visa 5 条全 Bad、e-Visa 全 Bad 等模式应作为**软性提示**写入 §10.3 prompt（非硬规则，因为存在小概率例外）。

### 12.1 目的与已确认范围

### 12.2 调研触发规则（ROI 驱动）

> **重要变更：** MF 线索不再一刀切「必做标准调研」。基于历史数据洞察（见 §13.2），Corporate Visa Verification 的核心动作是「转给 Willem 验证」而非「个性化拉新」，给这类线索做 LinkedIn 调研属于浪费。下表按线索类型 + 签证类别 + 评分综合判断，预期可砍掉约 40% 的调研成本。

| 评分 | 线索类型 / 签证特征 | 调研深度 | 说明 |
|------|------------------|---------|------|
| **GD** | 全部 | **全量**（LinkedIn + 公司 + News + CIPC） | 价值最高，必投 |
| **MF** | Individual + 签证 ∈ Bucket A（高净值类） | **标准**（LinkedIn + 公司） | 个性化拉新有 ROI |
| **MF** | Corporate + 签证 = Visa Verification | **跳过**（直接路由 Willem） | 操作模式确定，调研无价值 |
| **MF** | 其他（如 Individual + Bucket B/C、Corporate + 非 Verification） | **轻量**（仅 Web Search） | 验证背景，不做深度个人画像 |
| **MD** | 评分歧义、需边界判断 | **可选轻量**（仅 LinkedIn） | 仅在置信度低时触发 |
| **BD / Bad** | 全部 | **跳过** | 不致电，无需简报 |

**伪码：**

```
def determine_research_depth(lead):
    if lead.score == "GD":
        return "FULL"
    if lead.score == "MF":
        if lead.type == "Individual" and lead.visa_bucket == "A":
            return "STANDARD"
        if lead.type == "Corporate" and lead.visa_category == "Visa Verification":
            return "SKIP_ROUTE_TO_WILLEM"
        return "LIGHT_WEB_SEARCH"
    if lead.score == "MD" and lead.score_confidence == "low":
        return "LIGHT_LINKEDIN"
    return "SKIP"
```

### 13.5 验证流程（MVP 上线前必跑）

```
1. 取 §13.3 全部 30 条 few-shot 样本注入 §10.3 评分 prompt
2. 在 §13.4 全部 40 条回归集上跑评分（自动化脚本）
3. 计算：
   - 整体准确率 = (AI 评分 == 历史评分) / 40
   - 各级别 recall：GD recall、Bad recall（漏判 Bad 代价最高，应 ≥95%）
   - DNQ fixture 命中率 = §10.2 前置函数正确命中 §4.1 六条硬规则 / DNQ fixture 总数
   - DNQ 误杀率 = 被错误判为 DNQ 的非 DNQ 与 unknown 样本 / 非 DNQ 与 unknown fixture 总数
   - 软风险召回率 = apply_risk_flags 正确标记 §4.2 两条软风险 / soft-risk fixture 总数
4. 准确率 < 85% → 不允许 release，回到 prompt 调整
5. DNQ fixture、非 DNQ/unknown 防误杀 fixture 或软风险 fixture 未通过 → 不允许 release，补充字段提取、数据校验或规则实现
6. 上线后每月用新增线索扩充测试集，季度重训 few-shot
```

### 13.6 数据局限性说明（评审建议补充）

- **样本量小**：202 条对 LLM 评分而言偏少，特别是 BD（1 条）和 MF（20 条）。
- **GD 偏 RPV**：68 条 GD 中 50 条是 Retired Person Visa，模型可能过拟合此类。建议上线后定向收集其他 Bucket A 签证的 GD 样本。
- **历史标注一致性未知**：客户内部标注是否由同一人完成？是否存在评级漂移？建议项目启动会上向 Marisa/Melissa 核实。
- **时间维度缺失**：Excel 未含日期字段，无法判断模式是否随时间漂移（如 DHA 政策变更）。
